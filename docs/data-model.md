## Overview

All arithmetic logic and UI state are managed in React component state (and optionally browser `localStorage`). The data model described here covers the **optional server-side extension** for persisting calculation history.

---

## Architecture Decision

| Concern              | Approach                              |
|----------------------|---------------------------------------|
| Arithmetic logic     | Client-side JavaScript / React state  |
| Current display      | React `useState`                      |
| Calculation history  | `localStorage` (client, optional)     |
| Server persistence   | PostgreSQL via Prisma (optional)      |
| Authentication       | Not required for MVP                  |

## Client-Side State Model

For reference, the React state shape used in the calculator UI:

interface CalculatorState {
  display: string        // Current number shown on screen
  expression: string     // Full expression being built
  operator: string | null // Pending operator (+, -, *, /)
  waitingForOperand: boolean // True after operator is pressed
  history: HistoryEntry[] // Optional local history array
}

interface HistoryEntry {
  expression: string
  result: string
  timestamp: number // Date.now()
}

## Notes for Developers

1. **No migration needed for MVP** — the app works out of the box without running any SQL.
2. **localStorage key**: Use `calculator_history` as the localStorage key for consistency.
3. **Result precision**: Store results as strings to preserve display formatting (e.g., `"333.3333..."` for repeating decimals).
4. **Future extension**: To add user accounts, add a `userId` foreign key to `CalculationHistory` referencing a `User` table.
